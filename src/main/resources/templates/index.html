Why Jactl is safe to embed

Most embedded languages (Groovy, JS/Nashorn, BeanShell) expose the whole JVM and make you block dangerous things. Jactl inverts this: scripts start with nothing and the host opts capabilities in.

Blocked by the language itself

  • noJava classes & reflection — System.exit(), Runtime.exec() simply don't exist
  • noFile / network I/O
  • noThread creation or synchronization primitives
  • noAccess to host state except what's passed in

Try it: run System.exit(0) or new java.io.File('/') — both are compile errors, not runtime checks that could be bypassed.

What the host opts in

  • yesGlobals — this server injects a request map into every run (see the Globals example)
  • yesRegistered functions — this server registers serverInfo() and rot13(text) at startup; scripts can call those and nothing else

Host-level guards (this server)

  • 5s timeout — scripts run on a worker pool and are cancelled if they exceed it. The language can't prevent while(true){}, so the host must. Try the "Sandbox: infinite loop" example.
  • 64KB output cap — a println flood is truncated instead of exhausting memory.
  • Per-request globals — each run gets fresh state; scripts can't see each other.

Why this matters

This combination — capability-based language + host timeouts — is what makes Jactl suitable for untrusted user scripts: rule engines, customization hooks, report formulas, workflow conditions. Jactl also compiles to JVM bytecode (not interpreted), so hot scripts run at near-Java speed.

Suspend / resume

Jactl's party trick: built-in continuations. A registered function can do async work (DB call, HTTP) and the script suspends without blocking a thread, then resumes — scripts stay simple and synchronous-looking while the host stays non-blocking.
